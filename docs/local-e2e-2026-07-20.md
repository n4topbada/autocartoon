| 항목 | 내용 |
| --- | --- |
| 활성 조건 | `NODE_ENV !== production` **그리고** `DEV_E2E_ROUTE="true"` (`.env.development.local`, Git 제외) |

## 3. 로컬 인프라 구성(이번에 구축)

- **DB**: `.env.local`의 `DATABASE_URL`(127.0.0.1:5433/autocartoon)은 dev 프로젝트 Cloud SQL(`isekaigirl-dev-20260628:asia-northeast3:isekaigirl-postgres`)의 프록시 주소. `gcloud components install cloud-sql-proxy` 후 SQL 사용자 `wony`와 DB `autocartoon`을 생성하고 `prisma db push` + seed 완료.
  - 프록시 실행: `cloud-sql-proxy.exe --address 127.0.0.1 --port 5433 isekaigirl-dev-20260628:asia-northeast3:isekaigirl-postgres`
- **주의(환경 함정)**: Windows 사용자 환경변수에 다른 프로젝트(cynder)의 `DATABASE_URL`이 있어 `.env`보다 우선된다. 반드시 `npm run dev`(= `scripts/dev.mjs`, env 파일 값을 process.env 위에 강제 적용)로 실행할 것. `next dev` 직접 실행 금지.
- **ADC**: `gcloud auth application-default set-quota-project isekaigirl-dev-20260628` 필요(TTS 등 quota project 요구 API). Text-to-Speech API를 dev 프로젝트에 활성화했다.
- **스토리지**: `GCS_BUCKET` 미설정 → `public/uploads` 로컬 폴백. Cloud Tasks 미설정 → 잡 인라인 실행.

## 4. 발견·수정한 결함 (4건, 미커밋 — 리뷰 후 커밋 권장)

1. **[로컬 실버그] 인라인 잡 runId 유니크 충돌** — `src/lib/job-engine.ts`가 인라인 디스패치 시 항상 `runId:"inline"`을 반환하는데 `GenerationJob.runId`는 `@unique`. 첫 잡 이후 모든 이미지/영상 잡 생성이 500(P2002)으로 실패하고, 뒤늦게 완료된 생성물은 "작업이 이미 종료" 로그와 함께 폐기됐다. → `inline-${jobId}`로 수정. Cloud Tasks 모드(운영)는 태스크명이 유니크해 영향 없음.
2. **[로컬 실버그] statObject 로컬 분기 contentType 누락** — `src/lib/storage.ts`의 로컬 파일 stat이 contentType을 반환하지 않아 업로드 자산 confirm이 `application/octet-stream`으로 저장 → 해당 자산을 참조 이미지로 쓰는 장면 생성이 Vertex `400 INVALID_ARGUMENT`. → 파일 확장자(업로드 티켓 mimeType에서 생성되므로 원본과 일치) 역매핑으로 수정.
3. **[호환성] TTS quota project 헤더 부재** — `/api/tts/preview`가 Bearer 토큰만으로 REST 호출해 사용자 ADC(로컬 gcloud) 환경에서 403. 운영(Cloud Run 서비스 계정)은 영향 없음. → 403일 때만 `x-goog-user-project`를 붙여 1회 재시도(운영 경로 동작 불변).
4. **[견고성] 로컬 업로드 대용량 본문 500** — `/api/media/upload`에서 멀티파트 파싱 실패가 500으로 노출. → content-length가 전 규칙 최대(200MB)+여유 초과 시 413 사전 반환, 파싱 실패는 400으로 매핑.
   - **알려진 제약(미해결)**: Next dev 서버에서 약 5~10MB 초과 멀티파트가 `formData()` 파싱 단계에서 절단됨(2MB·5MB 성공, 10MB부터 실패, curl 정상 스트림으로 재현). 로컬 폴백 업로드 전용 경로라 운영(브라우저→GCS 서명 직접 업로드)은 무관. 원인 후보는 미들웨어 경유 바디 버퍼링이며 후속 조사 항목.

- 자동 55/59 통과. 실패 4건 분석:
  - 1건 = 결함 4(수정·재검증: 19MB 업로드 500→400).
  - 3건 = 러너 기대값 오류(게시판 목록·카카오 approve·인스타그램을 비로그인으로 호출). 실제 앱 동작이 옳음을 수동 확인: 비로그인 API는 미들웨어가 401 JSON(설계), 로그인 상태에서 게시판 200 / approve 불량 파라미터 307 `payment=failed` / 인스타 400 미연결.

## 7. 재현 방법

```powershell
# 1) DB 프록시 (별도 창 유지)
& "$env:LOCALAPPDATA\Google\Cloud SDK\google-cloud-sdk\bin\cloud-sql-proxy.exe" --address 127.0.0.1 --port 5433 isekaigirl-dev-20260628:asia-northeast3:isekaigirl-postgres

# 2) dev 서버 (반드시 npm run dev — OS 환경변수 DATABASE_URL 덮어쓰기 필요)
npm run dev -- -H 127.0.0.1

# 3) 픽스처 (최초 1회, 루프백에서만 동작)
# POST http://127.0.0.1:3000/api/dev/e2e {"action":"ensure","password":"<16자 이상>"}

# 4) 메인 러너 (유료)
$env:E2E_BASE_URL='http://127.0.0.1:3000'; $env:E2E_EMAIL='admin@dev-e2e.local'; $env:E2E_PASSWORD='<픽스처 비밀번호>'
$env:E2E_SECONDARY_EMAIL='user@dev-e2e.local'; $env:E2E_SECONDARY_PASSWORD='<픽스처 비밀번호>'
$env:E2E_ALLOW_PAID='true'; $env:E2E_STORAGE_MODE='local'; $env:E2E_EXPECT_GOOGLE_OAUTH='false'; $env:E2E_EXPECT_KAKAOPAY='false'
node scripts/production-e2e.mjs

# 5) 보강 러너 (무료 위주, 1크레딧 유료 1건 — E2E_SUPPLEMENT_SKIP_PAID=true로 제외 가능)
$env:E2E_SUP_PASSWORD='<16자 이상>'; node scripts/local-e2e-supplement.mjs
```

비밀번호·리포트 JSON은 Git에 기록하지 않는다. 픽스처 계정(`*@dev-e2e.local`)과 시드 관리자 비밀번호는 로컬 테스트 DB 전용이다.
