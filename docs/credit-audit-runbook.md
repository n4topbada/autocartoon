# 크레딧 감사 기록 운영 가이드

## 목적

크레딧 잔액 원장(`CreditLedger`)은 실제로 완료된 잔액 변경만 보존한다. 별도 감사 이벤트(`CreditAuditEvent`)는 성공, 거절, 외부 AI 실패, 결제 검증 실패, 자동 환불과 중복 요청 재사용까지 기록한다. 원장의 회계 의미를 흐리지 않으면서 운영자가 한 요청의 전체 흐름을 역추적하기 위한 구조다.

관리자는 `/admin`의 **크레딧 감사 기록**에서 최근 이벤트를 확인한다. 기본 기간은 7일이며 계정명, 이름, `CR-` 추적 코드, 생성 작업 ID, 결제 ID, 원장 ID와 오류 코드로 검색할 수 있다.

## 기록 범위

- AI·OCR·TTS·영상·캐릭터 설계의 선차감, 실행 실패, 전액 자동 환불
- 다중 이미지 일부 생성 실패와 미생성분 부분 환불
- 카카오페이 결제 시작, 검증, 실패·취소, 최종 크레딧 적립
- 쿠폰 신규 지급, 중복 요청 재확인, 만료·소진·잘못된 코드
- 관리자 수동 지급과 지급 실패
- 마켓 캐릭터 구매 차감과 잔액 부족
- Google·카카오 신규 가입 100크레딧
- OAuth 계정 연결 및 회원 탈퇴에 따른 잔액 정리

스키마 변경 시 기존 `CreditLedger`도 성공 이벤트로 한 번 역기록된다. 역사 데이터는 `HISTORICAL_LEDGER_BACKFILL` 사유 코드로 구분한다.

## 잔액 검증

각 이벤트에는 `balanceBefore`, `balanceAfter`, `units`, `direction`, `balanceVerified`가 있다.

- 성공 차감: `변경 후 = 변경 전 - 수량`
- 성공 지급·환불: `변경 후 = 변경 전 + 수량`
- 실패·상태 확인: 잔액이 바뀌지 않아야 함
- 계산이 맞지 않으면 `balanceVerified=false`로 저장되고 관리자 요약의 **잔액 불일치**에 집계됨

실제 잔액 변경, 성공 원장, 감사 이벤트는 같은 DB 트랜잭션에 기록한다. 감사 이벤트 저장이 실패하면 잔액 변경도 커밋되지 않는 것이 기본 원칙이다. 이미 롤백된 예외를 기록할 때는 별도 안전 기록 함수를 사용하고, 그 기록조차 실패하면 Cloud Run 구조화 오류 로그에 추적 코드를 남긴다.

## 장애 역추적 절차

1. 사용자에게 API 응답 또는 오류 화면의 `CR-XXXXXXXXXXXX` 추적 코드를 받는다. 생성 작업이면 작업 ID만 받아도 된다.
2. `/admin` → **크레딧 감사 기록** 검색창에 코드를 붙여 넣는다.
3. 시간순으로 `차감 성공 → 기능 실패 → 환불 성공` 또는 `결제 시작 → 검증 → 적립` 흐름을 확인한다.
4. 실패 행을 열어 오류 코드, 안전하게 정리된 오류 메시지, 제공자·모델, 작업·결제 참조를 확인한다.
5. **같은 흐름만 보기**를 눌러 동일 추적 코드의 이벤트만 남긴다.
6. 잔액 불일치가 있으면 원장 ID와 변경 전·후 값을 기준으로 수동 조정 전에 원인을 먼저 확인한다.

## 개인정보·민감정보

감사 메타데이터에는 프롬프트, 이미지 원문·base64, 쿠키, 인증 헤더, 비밀번호, API 키와 토큰을 저장하지 않는다. 공통 기록 계층이 민감한 키와 긴 불투명 문자열을 다시 가린다. 오류 메시지는 길이를 제한하고 토큰·서명 쿼리 값을 제거한다.

## 운영 기준

- 실패는 관리자 UI에서 빨간색, 성공은 검은색 상태로 표시한다.
- 감사 이벤트는 수정하지 않는 추가 전용 데이터로 취급한다.
- 현재 프로토타입에서는 삭제 주기를 두지 않는다. 사용량이 증가하면 12개월 온라인 보관 후 GCS/BigQuery 보관 정책을 검토한다.
- 수동 잔액 보정 기능을 추가할 때도 반드시 공통 `createCreditLedgerWithAudit` 경로를 사용한다.
